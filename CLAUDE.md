## Commands

```bash
# Development (runs Next.js + WebSocket server concurrently)
npm run dev

# Run only the WebSocket server
npm run dev:ws

# Run only Next.js
npm run dev:next

# Build
npm run build

# Lint
npm run lint

# Tests (vitest)
npm test

# Run a single test file
npx vitest run tests/unit/libs/apiKeyEncryption.test.ts

# Database migrations (Drizzle)
npx drizzle-kit generate
npx drizzle-kit migrate

# Test Redis connection
npm run test:redis
```

Environment variables are loaded from `.env.local`. Required vars include `DATABASE_URL`, `NEXT_PUBLIC_API_URL`, JWT config, and encryption keys.

## Architecture

This is a **SaaS AI chat app** (Next.js 15 + a standalone WebSocket server) that lets users bring their own LLM API keys.

### Two-Server Model

- **Next.js app** (`src/`) — handles UI, REST API routes (`src/app/api/`), and auth
- **WebSocket server** (`server/server.ts`) — handles real-time chat streaming; runs as a separate process on `PORT`/`WS_PORT` (default 8080). In production, managed via pm2.

### Path Aliases (tsconfig)

- `@/*` → `./src/*` or `./server/*`
- `@/shared/*` → `./shared/*`
- `@/server/*` → `./server/*`

### Shared Code (`shared/`)

Code used by both the Next.js app and the WebSocket server:

- `shared/db/schema.ts` — Drizzle ORM schema (PostgreSQL). Tables: `users`, `chats`, `messages`, `summaries`, `api_keys`, `refresh_tokens`, `access_codes`, `token_usage`
- `shared/lib/types.ts` — Core types (`Message`, `ChatContext`, `LLMProvider`, `StateAnnotation` for LangGraph, etc.)
- `shared/lib/container.ts` — Dependency injection container (`AgentDeps`) wiring DB utils and chat context services together
- `shared/lib/langchain/` — LLM integration:
  - `llmFactory.ts` — Creates LangChain chat models (OpenAI, Groq, Anthropic) from user's encrypted API keys; caches instances in an LRU cache (15 min TTL, keyed by `userId-provider`)
  - `llmHandler.ts` — `askQuestion()` async generator that streams LangGraph agent responses, updates context and DB post-stream
  - `llmHelper.ts` — Workflow graph setup, summarization, token calculation, DB update utilities
  - `getOrCreateChatContext.ts` — Retrieves or initializes `ChatContext` from the in-memory manager
  - `prompts.ts` — LangChain prompt templates
- `shared/constants.ts` — Enums for login providers, LLM providers, verification types
- `shared/logger.ts` — Winston logger factory

### Chat Context Management

`ChatContextManager` (`src/lib/services/chatContextManager.ts`) is a singleton LRU cache (200 entries, 30 min TTL) that stores `ChatContext` in memory. Context includes recent messages + a rolling summary generated by a non-streaming summary model. Summarization runs asynchronously in the background to avoid blocking the response stream.

### Authentication

- JWT-based: short-lived `accessToken` + longer-lived `refreshToken` stored as HttpOnly cookies
- `src/middleware.ts` protects `/chat/*`, `/dashboard/*`, `/api-keys/*` — on `JWTExpired`, redirects to `/api/auth/refresh-token`
- `shared/lib/jwt.ts` — shared JWT validation (used by both middleware and WebSocket auth)
- API keys (for LLM providers) are AES-encrypted before storage in the `api_keys` table

### Frontend

- `src/context/` — React contexts: `AuthContext`, `ChatContext`, `AllChatsContext`, `ThemeContext`
- `src/hooks/useWebSocket.ts` — Manages WebSocket lifecycle; handles streaming token assembly client-side (accumulates `type: "token"` chunks by message ID, finalizes on `type: "done"`)
- `src/components/` — UI components; `Chatbox`, `ChatListPanel`, `ChatListItem`, `Sidebar` are the main chat UI components
- Theme is toggled via `ThemeContext`; initial theme set by `public/scripts/set-theme.js` to avoid flash

### Testing

Tests live in `tests/unit/`. Vitest is configured with `vite-tsconfig-paths` so `@/*` aliases work. Mock files are co-located at `src/lib/__mocks__/` and `shared/lib/langchain/__mocks__/`. Setup file is `tests/setupEnv.ts`.
